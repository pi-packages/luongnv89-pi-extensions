## Proposed Content Summaries

### `docs/DEVELOPMENT.md`

A developer-focused guide covering:

- **Project layout** — how `extensions/` and `themes/` map to Pi's agent directories.
- **Extension structure** — a minimal extension skeleton (package.json with `pi.extensions` field, main entry point exporting a default function).
- **Extension API** — documented signatures for `ExtensionAPI.on()`, `pi.registerCommand()`, `ctx.ui.setFooter()`, `ctx.getContextUsage()`, and `ctx.model` shape, based on the usage in `statusline-pi/index.ts`.
- **Theme schema** — reference to the JSON Schema at `theme-schema.json`, list of available color keys (`accent`, `border`, `success`, etc.), and tips for creating consistent themes (mapping `vars` to semantic `colors`, thinking-level colors).
- **npm scripts** — `npm run install-all`, `npm run install-extensions`, `npm run install-themes` for local testing.
- **Coding conventions** — TypeScript with `import type`, `node:` prefix for builtins, no `console.log` in extension code (use `ctx.ui.notify`), async handlers, error-resilient patterns (try/catch with fallback values).
- **Testing** — manual test workflow (run `npm run install-all`, reload Pi, verify `/statusline-pi` toggle).
- **Submitting** — fork → branch → commit → PR (link to CONTRIBUTING.md).

Date will be filled in at release time.

### `CONTRIBUTING.md`

Standard contributing guide:

- **Getting started** — how to fork, clone, set up for local development.
- **Adding an extension** — directory naming, required files (package.json + entry point), registering commands, managing state via `ExtensionContext`.
- **Adding a theme** — JSON structure, `name` field, required color keys, referencing the schema URL.
- **Code style** — TypeScript conventions, async/await over raw promises, error handling (catch exceptions silently with fallbacks), no logging to stdout.
- **Commit style** — conventional commits encouraged but not required; descriptive messages.
- **PR checklist** — verify install with `npm run install-all`, test commands in Pi, update README if adding a new item, keep PRs focused on one extension/theme.

### `CODE_OF_CONDUCT.md`

Contributor Covenant v2.1 template adapted for the project. Standard sections: Our Pledge, Our Standards, Enforcement Responsibilities, Scope, Enforcement (reporting to the maintainer), Attribution.

### `SECURITY.md`

Minimal security policy:

- **Reporting a vulnerability** — open a private GitHub issue or (optionally) email the maintainer.
- **Response expectations** — acknowledgment within 72 hours, fix timeline depends on severity.
- **Scope** — the `install.sh` script (runs arbitrary commands) and extension runtime code are the primary concern.
- No PGP key or complex disclosure process needed at this stage.

### `.github/ISSUE_TEMPLATE/bug_report.md`

YAML-frontmatter template with fields:
- **Describe the bug** — clear description.
- **To Reproduce** — steps, Pi commands used, extension/theme involved.
- **Expected behavior** — what should happen.
- **Screenshots** — optional.
- **Environment** — OS, Pi version (`/version` output), extension versions.
- **Additional context** — free text.

### `.github/ISSUE_TEMPLATE/feature_request.md`

YAML-frontmatter template with fields:
- **Is your feature request related to a problem?** — problem description.
- **Describe the solution you'd like** — proposed extension/theme/behavior.
- **Describe alternatives you've considered** — other approaches.
- **Additional context** — screenshots, links, references.

### `.github/PULL_REQUEST_TEMPLATE.md`

Standard PR template with sections:
- **Summary** — what this PR does.
- **Related issue** — link to GitHub issue if applicable.
- **Type of change** — new extension, new theme, bug fix, docs, etc.
- **Testing** — what was tested and how.
- **Checklist** — README updated? Code installs without errors? Commands tested in Pi? No secrets committed?
